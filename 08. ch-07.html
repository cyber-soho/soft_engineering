Software Testing

Lecture: Software Testing: From Components to Systems

PART 1: COMPONENT-LEVEL TESTING

[1.0] A Strategic Approach to Software Testing

Welcome to our lecture on software testing. Testing is a set of activities that must be planned in advance and conducted systematically. It is not something we do just at the end; it's a phase that runs parallel to the entire development process. The goal is to uncover errors. A good test is one that has a high probability of finding a yet-undiscovered error.

A common myth is that testing is a "safety net" to add quality. This is false. You cannot "test in" quality. Quality is built in through good design, solid engineering practices, and effective technical reviews. Testing is the process we use to confirm that quality and to find the defects that- inevitably- creep in.

Our strategy begins at the smallest level‚Äîthe component‚Äîand works its way outward to the complete system.

EXAMPLE: Task: Building a new custom-designed house.

Relation to Lecture: This example illustrates the concept of a testing strategy that starts small and grows. You wouldn't wait until the entire house is built to see if things work. You'd have a strategy.

  • First, you inspect the foundation before you build walls on it (component testing).
  • Then, you check the plumbing connections between the bathroom and the main line (integration testing).
  • Finally, you do a "walk-through" of the finished house with the owner to make sure it meets their original requirements, like "a light switch for the patio is next to the back door" (validation testing).

Just "testing" at the end by moving a family in and seeing what breaks is a recipe for disaster.

[1.1] Verification and Validation (V&V)

You will hear two key terms: Verification and Validation, often shortened to V&V. They are not the same thing.

Verification: Answers the question, "Are we building the product right?" This means we are checking that our work correctly implements the specific functions as they were designed. Does this sort() function actually sort the numbers? Does this calculate_tax() module match the technical specification we wrote for it?

Validation: Answers the question, "Are we building the right product?" This is a higher-level check against customer requirements. We might have verified that our calculate_tax() module works perfectly according to the spec, but validation asks if the spec itself was correct. Does the module charge the 5% tax the customer actually wanted, or did we build a perfect module for a 7% tax that was based on a requirements mistake?

EXAMPLE: Task: Building a new car engine.

Relation to Lecture: This shows the difference between V&V. Verification would be running tests on the fuel injector to ensure it delivers exactly 10ml of gasoline per cycle, just as the engineering blueprints (the design) specified. We are verifying it matches the design. Validation would be putting the perfectly verified engine into the car and asking, "Does this car now meet the customer's requirement of 'going from 0-60 in 8 seconds'?" The injector might be "right" according to the design, but the design itself might be "wrong" for meeting the customer's actual performance goals.

[1.2] Organizing for Testing: The Independent Test Group (ITG)

There is a psychological conflict when a developer tests their own code. A developer's mindset is constructive; they want to build and create. Testing, however, has a "destructive" mindset; you are actively trying to break the software to find its flaws. A developer may unconsciously avoid the test cases that they know will break their code.

To solve this, many large projects use an Independent Test Group (ITG). This is a team of testers separate from the development team. Their entire job is to find errors. This removes the conflict of interest and is highly effective at improving software quality because the ITG brings a fresh, skeptical perspective. The developer is still responsible for testing their own components first (unit testing), but the ITG handles the "official" integration and system-level tests.

EXAMPLE: Task: Writing a novel.

Relation to Lecture: This demonstrates the value of an independent "tester." As the author (the developer), you write a chapter. You re-read it and fix typos you see (unit testing). However, you are too close to the work. You know what the plot is supposed to mean, so you read "what you meant" instead of "what you wrote." You then give the chapter to a professional editor (the ITG). The editor doesn't care about your feelings; they are paid to find problems. They will find plot holes, character inconsistencies, and grammar errors that you completely missed because they have an independent, unbiased perspective.

[1.3] The Testing Process: From "In the Small" to "In the Large"

Our testing strategy follows a clear progression, moving from a narrow scope to a broad one.

  1. Unit Testing: Focuses on a single component (a function, a method, a class). We test it in isolation.
  2. Integration Testing: We assemble the unit-tested components and test the interfaces between them.
  3. Validation Testing: We test the complete, integrated system against the user's requirements.
  4. System Testing: We test the software in the context of its full environment (hardware, other software, networks).

This lecture will follow this same path, starting "in the small" with Unit Testing.

EXAMPLE: Task: Assembling a bicycle from a kit.

Relation to Lecture: This maps the testing process to a physical assembly.

  • Unit Testing: You check each component in isolation. You spin the wheel to see if the axle is smooth. You squeeze the brake handle to see if the cable pulls.
  • Integration Testing: You attach the brake cable to the brake calipers. You now test the interface: does squeezing the handle (component A) correctly pull the cable and cause the calipers (component B) to close?
  • Validation Testing: The bike is fully assembled. You check it against the user's requirements (the box art!): "Is it a complete, rideable bicycle? Does it look like the picture?"
  • System Testing: You take the bike on the road (its environment). How does it handle on pavement? On gravel? Does the 'system' (bike + rider + road) work?
[1.4] Unit Testing Fundamentals

Unit testing focuses verification on the smallest unit of software design: the component. This could be a single function or a method within a class. We test it in isolation from the rest of the program.

Unit testing focuses on:

  • The Interface: Does it receive data correctly? Does it return data in the correct format?
  • Local Data: Does the function's internal data (e.g., local variables) get managed correctly?
  • Control Paths: Are all if-else branches and loops tested?
  • Boundary Conditions: Does it handle the "edges" (e.g., 0, -1, max value) correctly?
  • Error Handling: Does it behave properly when it receives bad data?

EXAMPLE: Task: Testing a toaster's "timer" knob.

Relation to Lecture: This is a unit test of a single component (the knob). We aren't testing the heating elements or the spring-loaded pop-up mechanism yet. We are testing only the timer knob in isolation.

  • Interface: Does the knob send the correct electrical signal (e.g., 1V for 1 minute, 2V for 2 minutes) to the control board?
  • Local Data: Does the internal gear in the knob stop at "5 minutes" when turned there?
  • Control Paths: Can the knob be turned? Can it be turned off?
  • Boundary Conditions: What happens when you turn it to its "0" (off) position? What happens at its "Max" (10 minutes) position?
  • Error Handling: What if you try to turn it past "Max"? Does it break, or does it stop safely?
[1.5] Scaffolding: Drivers and Stubs

When you test a component in isolation, you have a problem: it's not a full program. If your component (e.g., calculate_tax()) is called by the main application and itself calls a database function, how do you test it?

We build temporary, "throw-away" code called scaffolding.

  • Drivers: A "driver" is a piece of code that calls your component. It's a fake "main program." It "drives" the test, providing the test data (e.g., "Here, calculate_tax(), test yourself with an input of 100").
  • Stubs: A "stub" is a piece of code that is called by your component. It's a fake "database" or "other module." If calculate_tax() needs to call get_tax_rate_from_database(), you write a stub for that function that just returns a fixed value (e.g., "return 0.05;") without actually touching a database.

EXAMPLE: Task: Testing a car's transmission.

Relation to Lecture: This demonstrates drivers and stubs. You can't test a new transmission just by holding it in your hands. You need to put it on a test rig.

  • Driver: The test rig's engine that connects to the transmission's input shaft is the driver. It's a fake "car" that provides the power and tells the transmission when to shift.
  • Stubs: The test rig's connectors that attach to the transmission's output shafts (where the wheels would go) are the stubs. They don't go anywhere, but they provide a place for the transmission to send its output power so it can be measured.
PYTHON EXAMPLE: Unit Test with Driver and Stubs
# --- The Component To Be Tested ---
# This is the function we want to test.
def get_total_price(item_id):
    # It calls another function to get the base price.
    base_price = get_base_price_from_db(item_id)
    # It calls another function to get the tax rate.
    tax_rate = get_current_tax_rate()
    # It returns the final price.
    return base_price * (1 + tax_rate)

# --- Scaffolding (Stubs) ---
# **This is a STUB.**
# It's a fake version of the database function.
def get_base_price_from_db(item_id):
    # It doesn't connect to a real database.
    # It just returns a fixed, predictable value for our test.
    print(f"STUB: get_base_price_from_db called with {item_id}")
    # We'll pretend item 'abc' costs 100.
    if item_id == 'abc':
        # Return 100.0
        return 100.0
    # Return 0.0 for any other item.
    return 0.0

# **This is another STUB.**
# It's a fake version of the tax rate function.
def get_current_tax_rate():
    # It just returns a fixed, predictable rate.
    print("STUB: get_current_tax_rate called")
    # Return a 10% tax rate.
    return 0.10

# --- Scaffolding (Driver) ---
# **This is the DRIVER.**
# It's our "main program" that runs the test.
def run_unit_test():
    # Announce the test.
    print("**DRIVER: Running test for get_total_price...**")
    # Define the item we want to test.
    test_item = 'abc'
    # Define the *expected* result. We know 100 * (1 + 0.10) = 110.
    expected_result = 110.0
    
    # Call the component under test.
    actual_result = get_total_price(test_item)
    
    # Announce the results.
    print(f"DRIVER: Expected={expected_result}, Actual={actual_result}")
    
    # Check if the test passed or failed.
    if actual_result == expected_result:
        # Print a success message.
        print("***Test PASSED!***")
    else:
        # Print a failure message.
        print("***Test FAILED!***")

# --- Execution ---
# This line runs our driver.
run_unit_test()


                    


                

            


            

                [1.6] White-Box Testing
                

                    
White-Box Testing (also called "glass-box" or "structural" testing) is a testing philosophy where the tester has full access to the internal logic and source code of a component. You are "looking inside the box."

                    
The goal is not just to see if the output is right, but to ensure that all the internal paths of the code are exercised. For example, if you have an if-else statement, you will design two tests: one that makes the if condition True, and one that makes it False.

                    
                    

                    

                        
EXAMPLE: Task: Testing a hotel's automatic sliding door.

                        
Relation to Lecture: This shows the white-box approach. A black-box tester would just walk at the door and see if it opens. A white-box tester is the engineer who built it. They have the wiring diagram (the source code). They will test specific internal logic:

                        
    
                            
  • "I will test the motion sensor (path 1) by waving my hand."
    • 
                            
  • "I will test the pressure-pad sensor (path 2) by stepping on the mat."
    • 
                            
  • "I will test the 'emergency open' button (path 3) by pressing it."
    • 
                        

                        
They are not just testing the function (door opens), but all the internal paths that cause it to open.

                    

                

            


            

                [1.7] White-Box: Basis Path Testing
                

                    
This is a specific, formal technique for white-box testing. It was developed by Tom McCabe and uses a concept called Cyclomatic Complexity (V(G)).

                    
In simple terms, Cyclomatic Complexity is a number that measures the logical complexity of a program. It tells you exactly how many "independent paths" exist in a piece of code. This number is the minimum number of test cases you need to guarantee that you have executed every single line of code at least once.

                    
You calculate it by:

                    
    
                        
  • Drawing a "flow graph" of the code (like a flowchart).
    • 
                        
  • Counting the number of "regions" in the graph (easiest way).
    • 
                        
  • Or, V(G) = E - N + 2 (where E = edges, N = nodes)
    • 
                        
  • Or, V(G) = P + 1 (where P = predicate nodes, i.e., if, while statements)
    • 
                    

                    
If a function has a V(G) of 4, you know you need to design 4 specific tests to cover all "basis paths."

                    
                    

                        
PYTHON EXAMPLE: Basis Path Testing & V(G)

                        
# --- The Component To Be Tested ---
# This function has a few logical paths.
def process_value(x):
    # A single entry point (node 1)
    print(f"Input: {x}")
    
    # A predicate node (P=1) (node 2)
    if x < 10:
        # Path 1
        print("Path 1")
    else:
        # Path 2
        print("Path 2")
    
    # Another predicate node (P=2) (node 3)
    if x > 50:
        # Path 3
        print("Path 3")
        
    # A single exit point (node 4)
    print("Done")

# --- Analysis ---
# This code has 2 predicate nodes ('if' statements).
# Using the formula V(G) = P + 1:
# V(G) = 2 + 1 = 3
#
# This tells us we need **3** basis path test cases
# to cover all independent paths.
#
# A possible basis set of paths is:
# Path A: (x < 10) is True, (x > 50) is False.  (e.g., x = 5)
# Path B: (x < 10) is False, (x > 50) is False. (e.g., x = 30)
# Path C: (x < 10) is False, (x > 50) is True.  (e.g., x = 100)
#
# (Note: Path (x < 10) True and (x > 50) True is impossible)

# --- Driver and Test Cases ---
# Announce the testing.
print("--- Basis Path Testing ---")
# **Test Case 1 (Path A: x = 5)**
# This covers "Path 1" and skips "Path 3".
print("\n**Test Case 1 (x=5):**")
# Call the component.
process_value(5)
# **Test Case 2 (Path B: x = 30)**
# This covers "Path 2" and skips "Path 3".
print("\n**Test Case 2 (x=30):**")
# Call the component.
process_value(30)
# **Test Case 3 (Path C: x = 100)**
# This covers "Path 2" and "Path 3".
print("\n**Test Case 3 (x=100):**")
# Call the component.
process_value(100)
# We have now exercised every `print` statement with 3 tests.


                        

                            
                            
                        

                        

                    


                    
                

            


            

                [1.8] White-Box: Control Structure Testing (Loop Testing)
                

                    
Loops (for, while) are a common source of errors. Loop testing is a white-box technique that focuses exclusively on their validity. For a simple loop, we should test:

                    
    
                        
  • Skip the loop: Test what happens if the loop never runs (e.g., the list is empty).
    • 
                        
  • One pass: Test the loop with only one item.
    • 
                        
  • Two passes: Test with two items (to check for re-initialization errors).
    • 
                        
  • m passes: Test with a "typical" number of items (where m < n, the max).
    • 
                        
  • n-1, n, n+1 passes: Test at the boundary of the loop (if it has a known max n).
    • 
                    

                    

                        
EXAMPLE: Task: Testing a new, high-tech elevator.

                        
Relation to Lecture: This demonstrates loop testing. The "loop" is the elevator's logic for processing its list of "floor requests."

                        
    
                            
  • Skip the loop: What happens if the elevator is at rest and no buttons are pressed? (It should do nothing).
    • 
                            
  • One pass: A person gets in on floor 1 and presses "3". (Test a single request).
    • 
                            
  • Two passes: A person on floor 1 presses "4", and another person on floor 2 presses "5". (Test a simple queue of requests).
    • 
                            
  • m passes: Test with 5-6 requests in the queue.
    • 
                            
  • n passes: Test with a full queue (the elevator's max, e.g., 10 requests). What happens if someone tries to add an 11th request? (This is the n+1 test).
    • 
                        

                    

                

            


            

                [1.9] Black-Box Testing
                

                    
Now we switch to the opposite philosophy. Black-Box Testing (also called "behavioral" or "functional" testing) treats the component as a "black box." The tester has no knowledge of the internal code or logic.

                    
The only thing the tester cares about is the requirements. They provide an input and check that the output is correct, according. They ask questions like:

                    
    
                        
  • "How is functional validity tested?"
    • 
                        
  • "What classes of input will make good test cases?"
    • 
                        
  • "Is the system sensitive to certain input values?"
    • 
                    

                    

                        
EXAMPLE: Task: Testing an ATM (Automated Teller Machine).

                        
Relation to Lecture: This is the classic black-box example. As a user (a black-box tester), you have no idea what programming language the ATM uses. You don't have its source code. You are "outside the box." You test it based on its expected behavior:

                        
    
                            
  • Test 1: I put in my card, type my PIN (1234), request $100. Expected Output: The machine gives me $100 and a receipt.
    • 
                            
  • Test 2: I put in my card, type a wrong PIN (9999), request $100. Expected Output: The machine shows an "Invalid PIN" error and gives no money.
    • 
                            
  • Test 3: I put in my card, type my PIN, request $1,000,000. Expected Output: The machine shows an "Insufficient Funds" error.
    • 
                        

                    

                

            


            

                [1.10] Black-Box: Equivalence Partitioning
                

                    
This is a black-box technique for reducing the number of test cases. If an input field accepts a number from 1 to 100, you don't need to test all 100 values. Instead, you "partition" the input domain into "equivalence classes" where all values are treated the same by the logic. For "1 to 100", the partitions are:

                    
    
                        
  • Invalid: < 1 (e.g., -5, 0)
    • 
                        
  • Valid: 1 to 100 (e.g., 50)
    • 
                        
  • Invalid: > 100 (e.g., 101, 500)
    • 
                    

                    
The theory is that a test with 50 will find the same bugs as a test with 30 or 70. You only need to pick one value from each partition. This gives you 3 powerful test cases (-5, 50, 500) instead of hundreds of weak ones.

                    
                    

                    

                        
EXAMPLE: Task: Testing a website's "Age" field for buying wine (must be 21 or older).

                        
Relation to Lecture: This shows how to partition input data. The partitions are:

                        
    
                            
  • Invalid (Underage): Age < 21. (We'll pick: 17)
    • 
                            
  • Valid (Of Age): Age >= 21. (We'll pick: 35)
    • 
                            
  • (We might also add partitions for absurd-but-possible inputs)
    • 
                            
  • Invalid (Absurd low): Age < 0. (We'll pick: -5)
    • 
                            
  • Invalid (Absurd high): Age > 130. (We'll pick: 200)
    • 
                            
  • Invalid (Type): Not a number. (We'll pick: "hello")
    • 
                        

                        
We now have 5 test cases (17, 35, -5, 200, "hello") that cover all logical partitions.

                    

                

            


            

                [1.11] Black-Box: Boundary Value Analysis (BVA)
                

                    
Boundary Value Analysis (BVA) is a partner to Equivalence Partitioning. It's based on the fact that a huge number of errors happen at the boundaries of an equivalence class. Programmers make "off-by-one" errors. They use < (less than) when they meant <= (less than or equal). So, for our "1 to 100" example, BVA says we must test at and just next to the boundaries:

                    
    
                        
  • Boundary 1 (Min): Test 0 (invalid), 1 (valid), 2 (valid).
    • 
                        
  • Boundary 2 (Max): Test 99 (valid), 100 (valid), 101 (invalid).
    • 
                    

                    
Combining BVA and Equivalence Partitioning is a very powerful way to design black-box tests.

                    
                    

                    

                        
PYTHON EXAMPLE: Finding a Bug with BVA

                        
# --- The Component To Be Tested ---
# This function has a bug! It uses '<' instead of '<='.
def is_discount_code_valid(code):
    # Check if the code is a number
    if not isinstance(code, int):
        # Path 1: Not an integer
        return False
    # Check if the code is in the valid range
    # **BUG HERE!** Should be `code >= 1000`
    if code > 1000 and code < 2000:
        # Path 2: Valid
        return True
    else:
        # Path 3: Out of range
        return False

# --- Driver (Test Cases using BVA) ---
# We will test the boundaries: 1000 and 2000
# Our test points will be: 999, 1000, 1001 and 1999, 2000, 2001
# Announce the testing.
print("--- Boundary Value Analysis (BVA) ---")

# A helper function to run tests
def run_bva_test(test_value, expected_result):
    # Run the test
    actual_result = is_discount_code_valid(test_value)
    # Check the result
    if actual_result == expected_result:
        # Pass
        print(f"Test {test_value}: PASSED (Expected: {expected_result})")
    else:
        # Fail
        print(f"***Test {test_value}: FAILED! (Expected: {expected_result}, Got: {actual_result})***")

# --- Test the 'min' boundary (1000) ---
# Test just below the boundary
run_bva_test(999, False)
# Test *at* the boundary
run_bva_test(1000, True)  # This test will FAIL and find our bug!
# Test just above the boundary
run_bva_test(1001, True)

# --- Test the 'max' boundary (2000) ---
# Test just below the boundary
run_bva_test(1999, True)
# Test *at* the boundary
run_bva_test(2000, True)  # This test will ALSO FAIL!
# Test just above the boundary
run_bva_test(2001, False)


                        

                            
                            
                        

                        

                    

                

            


            

                [1.12] Object-Oriented (OO) Testing: Class Testing
                

                    
When we move to Object-Oriented (OO) programming, the "unit" is no longer a single function. The smallest testable unit is the Class. We can't just test a single method (function) in isolation, because that method might depend on the state of the object (i.e., its internal data or attributes).

                    
For example, a BankAccount class might have a withdraw() method. Testing it "in isolation" makes no sense. We must first call the deposit() method to change the object's balance attribute. Class testing, therefore, involves testing sequences of operations that change the object's state.

                    

                        
EXAMPLE: Task: Testing the ShoppingCart class for an e-commerce site.

                        
Relation to Lecture: This shows how OO testing requires testing "state" and "sequences." We can't just test the checkout() method. We have to test a sequence of method calls that represents a "life history" of the object:

                        
Test Sequence 1 (Normal):

                        
    
                            
  1. cart = new ShoppingCart() (Test the constructor)
    2. 
                            
  2. cart.addItem("apple") (Test addItem)
    3. 
                            
  3. cart.addItem("orange")
    4. 
                            
  4. total = cart.getTotal() (Test getTotal. Should be > 0)
    5. 
                            
  5. cart.checkout() (Test checkout in a valid state)
    6. 
                        

                        
Test Sequence 2 (Error State):

                        
    
                            
  1. cart = new ShoppingCart()
    2. 
                            
  2. cart.checkout() (Test checkout when cart is empty. Should fail gracefully).
    3. 
                        

                        
Test Sequence 3 (Remove item):

                        
    
                            
  1. cart = new ShoppingCart()
    2. 
                            
  2. cart.addItem("apple")
    3. 
                            
  3. cart.removeItem("apple") (Test removeItem)
    4. 
                            
  4. total = cart.getTotal() (Test getTotal. Should be 0)
    5. 
                            
  5. cart.checkout() (Test checkout when cart is now empty)
    6.

                [2.0] Introduction to Integration Testing
                

                    
We have successfully "unit tested" all our components. They all work perfectly... in isolation. Now, we must assemble them. Integration testing is a systematic technique for building the software architecture while simultaneously testing the interfaces between components. The goal is to find errors that occur when components interact.

                    
Common integration errors include:

                    
    
                        
  • Data loss: One component passes data, but the receiving component misinterprets it (e.g., "Is that value in feet or meters?").
    • 
                        
  • Adverse effects: Component A works fine, but when it runs, it accidentally overwrites a piece of memory that Component B was using, causing B to fail.
    • 
                        
  • Interface mismatch: Component A passes three arguments, but Component B was updated to only expect two.
    • 
                    

                    
A "Big Bang" approach, where you combine everything at once and test, is a recipe for chaos. When a test fails, you have no idea which of the 100 new integrations caused the bug. We must use an incremental strategy.

                    

                        
EXAMPLE: Task: Building a car.

                        
Relation to Lecture: This shows the "Big Bang" failure vs. an incremental approach.

                        
    
                            
  • Big Bang: You build the engine, transmission, steering wheel, and brakes all separately and confirm they work (unit testing). Then, you bolt them all into the car at the same time. You turn the key, and the car lurches and crashes. What failed? Was it the engine-to-transmission connection? The steering-to-wheel connection? You have no idea.
    • 
                            
  • Incremental:
                                
      
                                    
    1. Connect just the engine to the transmission. Test that integration.
      2. 
                                    
    2. Then, connect the transmission to the wheels. Test that integration.
      3. 
                                    
    3. Then, connect the steering wheel to the steering rack. Test that.
      4. 
                                
    
                                This incremental approach lets you pinpoint failures at each new "interface."
                            
    • 
                        

                    

                

            


            

                [2.1] Top-Down Integration
                

                    
This is an incremental strategy that starts at the "top" of the program: the main control module (e.g., the main menu or the main() function).

                    
    
                        
  1. You start with the main module.
    2. 
                        
  2. All modules that main calls are replaced with Stubs.
    3. 
                        
  3. You test the main module's logic.
    4. 
                        
  4. One by one, you replace a Stub with the real unit-tested component.
    5. 
                        
  5. You re-run your tests to make sure the new integration works.
    6. 
                        
  6. You repeat this, moving downward through the program's control hierarchy.
    7. 
                    

                    
This approach is good because it tests the major control and decision-making logic of the application very early.

                    
                    

                    

                        
EXAMPLE: Task: Integrating a new fast-food "Point of Sale" (POS) system.

                        
Relation to Lecture: This shows a top-down integration.

                        
    
                            
  1. Start with the "Main Menu" screen (the top-level module).
    2. 
                            
  2. The "Take Order," "Process Payment," and "Kitchen Report" buttons are all just stubs. Clicking "Process Payment" just shows a message: "STUB: Payment module called."
    3. 
                            
  3. Test the Main Menu logic (e.g., can you navigate between fake buttons?).
    4. 
                            
  4. Integration Step 1: Replace the "Take Order" stub with the real "Order" module. Now you can test the real interaction between the Main Menu and the Order screen. The "Process Payment" button is still a stub.
    5. 
                            
  5. Integration Step 2: Replace the "Process Payment" stub with the real "Payment" module. Now you can test the full flow: Main Menu -> Order -> Payment.
    6. 
                        

                    

                

            


            

                [2.2] Bottom-Up Integration
                

                    
This is the opposite approach. It starts at the "bottom" with the lowest-level, most fundamental components (e.g., utility functions, database access modules).

                    
    
                        
  1. You group low-level components into "clusters" or "builds" that perform a specific sub-function.
    2. 
                        
  2. You use a Driver to call this cluster and test it.
    3. 
                        
  3. You gradually combine clusters, moving upward through the program structure.
    4. 
                        
  4. Drivers are removed as higher-level modules are added (since the higher-level modules become the new "drivers" for the cluster).
    5.

                [2.3] Continuous Integration (CI) and Smoke Testing
                

                    
Continuous Integration (CI) is a common practice in agile and DevOps teams. It's the practice of merging all developers' work into a central "mainline" branch continuously‚Äîoften multiple times per day. Each merge triggers an automated build and test.

                    
Smoke Testing is the test that is run every single time a new build is created (e.g., by the CI system).

                    
    
                        
  • It is a shallow and wide test, not a deep one.
    • 
                        
  • It's designed to answer one question: "Did this new change break the build so badly that we can't even test it?"
    • 
                        
  • It exercises the entire system from end-to-end, but only for its most critical, high-level functions.
    • 
                        
  • If the smoke test fails, the build is immediately rejected, and no more testing is done until it's fixed. It's called a smoke test because, in old hardware, if you turned it on and it didn't smoke, it was "good to go" for more testing.
    •

Relation to Lecture: This shows CI and smoke testing in action.

                        
    
                            
  • Without CI: All 10 developers work in isolation for a month. On "Integration Day," they all try to merge their code. It's chaos. Nothing works.
    • 
                            
  • With CI:
                                
      
                                    
    1. Developer A finishes a small new feature (e.g., changing a button color).
      2. 
                                    
    2. She commits her change to the central repository.
      3. 
                                    
    3. The CI Server (like Jenkins or GitHub Actions) automatically detects this change.
      4. 
                                    
    4. It builds the entire website.
      5. 
                                    
    5. It runs the Smoke Test. The smoke test might just:
                                        
        
                                            
      • Load the Homepage.
        • 
                                            
      • Try to log in with a test account.
        • 
                                            
      • Visit the "Products" page.
        • 
                                            
      • Log out.
        • 
                                        
      
                                    
      6. 
                                    
    6. This whole process takes 60 seconds. The test passes. The build is "green."
      7. 
                                    
    7. Developer B now commits his change. The CI server builds and runs the smoke test... which FAILS. The login page is broken.
      8. 
                                    
    8. The build is "red." The entire team is notified. Developer B immediately fixes the bug before it causes problems for anyone else.
      9. 
                                
    
                            
    • 
                        

                    

                

            


            

                [2.4] Regression Testing
                

                    
Each time you add a new component or fix a bug, you create a risk. You might have unintentionally broken something else that used to work. This is called a "regression."

                    
Regression Testing is the re-execution of a subset of tests that have already been conducted, to ensure that changes have not introduced new, unintended side effects. As you build your test suite, you "tag" certain tests as "regression tests." These are the most critical, end-to-end tests. Every time a major change is made, this "regression suite" is run to ensure the core functionality is still intact. This is often automated.

                    
                    

                        
PYTHON EXAMPLE: Catching a Regression

                        
# --- The Original Component ---
# This class worked perfectly.
class Calculator_v1:
    # The add method.
    def add(self, a, b):
        # Return the sum.
        return a + b

# --- The "Regression Test Suite" ---
# This test was written for v1.
def run_regression_suite(calculator_instance):
    # Announce the test.
    print(f"\n--- Running REGRESSION suite on {type(calculator_instance).__name__} ---")
    # Define the test.
    expected = 5
    # Run the add test.
    actual = calculator_instance.add(2, 3)
    # Check the result.
    if actual == expected:
        # Pass
        print("REGRESSION TEST (add): PASSED")
        # Return True for success.
        return True
    else:
        # Fail
        print(f"***REGRESSION TEST (add): FAILED! (Expected {expected}, Got {actual})***")
        # Return False for failure.
        return False

# --- The New Component (v2) ---
# A developer adds a new 'multiply' method...
# ...and *accidentally* introduces a bug in 'add'.
class Calculator_v2:
    # The add method.
    def add(self, a, b):
        # ***Accidental Bug!*** The developer broke this.
        return a + b + 1  # Oops!
    
    # The new multiply method.
    def multiply(self, a, b):
        # Return the product.
        return a * b

# --- Driver (Test Execution) ---
# Create an instance of the *old* version.
calc_v1 = Calculator_v1()
# Run regression tests on v1. It passes.
run_regression_suite(calc_v1)

# Now, the developer finishes v2.
calc_v2 = Calculator_v2()
# We run the *same regression test* on the *new version*.
# **This test will catch the regression!**
run_regression_suite(calc_v2)

                [3.1] Alpha and Beta Testing
                

                    
Validation testing is often done by the development team or ITG. But for software products built for a wide audience, there's a final step: Acceptance Testing. This is where the customer tests the software. This is often done in two phases:

                    
    
                        
  • Alpha Test: This is conducted at the developer's site (in a controlled environment). The "customers" (a select group of real users) are invited to come in and use the software. The developers "look over their shoulder," watch how they use it, and record any errors or usage problems.
    • 
                        
  • Beta Test: This is conducted at the end-user's sites (in the "wild"). The developer is not present. The software is given to a wider group of external users (beta testers) who use it for their real, day-to-day work. They report bugs back to the developer. This is a "live" test in an uncontrolled environment and is invaluable for finding bugs related to real-world hardware, network conditions, and user behaviors.
    • 
                    


                    

                    

                        
EXAMPLE: Task: Releasing a new video game.

  • Beta Test: The company releases a "Closed Beta" or "Open Beta" version of the game on the internet. 500,000 people download it and play it at home on their own PCs and own internet connections.                 [3.2] Specialized Testing for Mobility
                

                    
When we test mobile applications (MobileApps), we have a new, unique set of challenges. The testing strategy must be specialized. Key issues include:

    • 
                        
  • Context: Mobile apps are "context-aware." They use GPS (Global Positioning System), the camera, the accelerometer. These must be tested.

                [3.8] Security Testing
                

                    
Security testing is a specialized test that attempts to verify that protection mechanisms built into a system actually work. The tester adopts the mindset of an attacker. Security testing probes for vulnerabilities like:

                    
    
                        
  • Denial-of-Service (DoS): Can I "flood" your app with so much traffic that it crashes?
    • 
                        
  • Data Theft: Can I "guess" the URL for another user's "My Account" page (.../account/123) and see their data? (This is called an "Insecure Direct Object Reference").
    • 
                        
  • Injection Attacks: If a form asks for my "Username," what if I type in a "SQL (Structured Query Language)" command instead? (' OR 1=1 --). Will the system execute my command and log me in as an administrator?
    • 
                        
  • Cross-Site Scripting (XSS): If I make a forum post, can I include malicious <script> tags that will run in the browser of other users who read my post?
    •

                [3.9] Performance Testing
                

                    
Performance testing is designed to test the run-time performance of software. It answers questions like, "Is it fast?"

                    
    
                        
  • Response Time: How long does it take for the page to load after a user clicks a link?
    • 
                        
  • Resource Utilization: How much CPU (Central Processing Unit) or memory does the app use? (On mobile, how much battery does it use?)
    • 
                        
  • Throughput: How many transactions can the system handle per second?
    • 
                    

                    
Performance testing is not focused on finding functional bugs, but on finding bottlenecks.

                    

                        
EXAMPLE: Task: Testing a "stock trading" application.

                        
Relation to Lecture: This shows why performance is a "non-functional" but critical requirement.

    • 
                            
  • Performance Test: Click "Buy 100 shares of XYZ." Result: It took 30 seconds for the "Confirm" page to load. (FAIL)
    • 
                        

                        
In the world of stock trading, a 30-second delay is a catastrophic failure. The function worked, but the performance was unacceptable. The performance test's goal is to find what caused that 30-second delay (e.g., a slow database query).

                    

                

            


            

                [3.10] Load Testing and Stress Testing
                

                    
These are two specific types of performance testing.

                    
    
                        
  • Load Testing: Tests the system under expected, realistic loads. The goal is to see how the system behaves under normal and peak conditions. (e.g., "How does the server handle 1,000 concurrent users, which is our normal peak traffic?").
    • 
                        
  • Stress Testing: Tests the system beyond its expected limits. The goal is to break the system. We want to find the "breaking point" and, more importantly, what happens when it breaks. Does it crash and burn? Or does it degrade gracefully? (e.g., "Let's hit the server with 50,000 concurrent users. I know it will fail, but I want to see how. Does it show a 'Server Busy, please try again' page, or does it crash the database?").
    • 
                    

                    

                        
EXAMPLE: Task: Testing an e-commerce website before Black Friday.

                        
Relation to Lecture: This shows the difference between Load and Stress testing.

                        
    
                            
  • Load Test: You expect 10,000 concurrent users at 9:00 AM on Black Friday. So, you run a test that simulates 10,000 users. You measure the response time. It's 2 seconds. The team decides this is "acceptable."
    • 
                            
  • Stress Test: You want to find the breaking point. You start a test at 1,000 users and increase it... 10,000... 20,000... 30,000... At 35,000 users, the server response time hits 15 seconds. At 40,000 users, the server crashes.
    • 
                        

                        
Result: The breaking point is ~35,000 users.

                        
Graceful Degradation: When it crashed, did it take the whole database with it? Or did it just stop accepting new connections? The "Stress Test" answers this.

                    

                

            


            

                [3.11] Testing Artificial Intelligence (AI) Systems
                

                    
Testing AI (Artificial Intelligence) and Machine Learning (ML) systems is very different.

                    
    
                        
  • You can't use traditional white-box testing. You can't check the "logic" of a neural network with 10 billion parameters.
    • 
                        
  • The system is "non-deterministic." It gives probabilities, not simple "right" or "wrong" answers.
    • 
                        
  • The "source code" is also the "data." The system's behavior is defined by the data it was trained on.
    • 
                    

                    
Testing focuses on:

                    
    
                        
  • Data Validation: Is the training data biased? Does it have errors?
    • 
                        
  • Model Validation: How accurate is the model? (e.g., "It correctly identifies 95% of cat pictures").
    • 
                        
  • Robustness: What happens when you give it "adversarial" or "nonsense" input?
    • 
                    

                    

                        
EXAMPLE: Task: Testing an AI "Spam Filter" for email.

                        
Relation to Lecture: This shows the non-deterministic, data-driven nature of AI testing.

    • 
                            
  • AI Test:
                                
      
                                    
    1. Get a "test set" of 10,000 emails the model has never seen before. This set has been hand-labeled by humans (5,000 spam, 5,000 not-spam).
      2. 
                                    
    2. Run all 10,000 emails through the AI model.
      3. 
                                    
    3. Measure the results:

                [3.12] Accessibility Testing
                

                    
Accessibility Testing (often abbreviated as "a11y") is the process of verifying that all people can use the application, regardless of disability. This includes users who are:

                    
    
                        
  • Visually impaired (blind, low-vision, color-blind)
    • 
                        
  • Hearing impaired
    • 
                        
  • Mobility impaired (cannot use a mouse)
    • 
                        
  • Cognitively impaired
    • 
                    

                    
This testing involves:

                    
    
                        
  • Screen Readers: Can a blind user navigate the entire site using a screen reader (like JAWS or NVDA)? Is every button labeled? Is every image given "alt text"?
    • 
                        
  • Keyboard Navigation: Can you use the entire site using only the "Tab" and "Enter" keys? Or are there "mouse-only" traps?
    • 
                        
  • Color Contrast: Is the "light grey text on a white background" readable for a low-vision user? (There are tools to check this).
    • 
                        
  • Captions: Are all videos captioned for hearing-impaired users?
    • 
                    

                    

                        
EXAMPLE: Task: Testing a website's "Buy Now" button.

                        
Relation to Lecture: This is an accessibility test.

                        
    
                            
  • Sighted User: Sees a big, green "Buy Now" button. No problem.
    • 
                            
  • Accessibility Test 1 (Screen Reader): A blind user navigates to the button.
                                
      
                                    
    • Failure: The screen reader says "Button." (The user has no idea what this button does).
      • 
                                    
    • Success: The screen reader says "Button: Buy Now, $19.99." (The aria-label attribute was set correctly in the HTML).
      • 
                                
    
                            
    • 
                            
  • Accessibility Test 2 (Keyboard): A user who cannot use a mouse presses the "Tab" key.
                                
      
                                    
    • Failure: The "Tab" focus skips the button entirely. The user cannot purchase the item.
      • 
                                    
    • Success: The "Tab" key puts a visible "focus ring" around the button, and pressing "Enter" activates it.
      • 
                                
    
                            
    • 
                        

                    

                

            


            

                [3.13] Documentation and Help Facility Testing
                

                    
Errors in documentation are as bad as errors in code. Nothing is more frustrating than following a "Help" guide exactly and having it fail. Documentation testing must be part of the test plan.

                    
    
                        
  • Technical Review: Is the document clear? Are there typos?
    • 
                        
  • Live Test: Use the documentation with the actual program.
                            
      
                                
    • Follow the "How To" guide step-by-step. Does it work? Are the screenshots up-to-date?
      • 
                                
    • When the program shows an error message (e.g., "Error 5001"), does the documentation have an entry for "Error 5001"?
      • 
                                
    • Is the Table of Contents and Index accurate?
      • 
                            
    
                        
    •
