Topic 1: Foundations & Overview

This topic covers the fundamental nature of software, the principles of engineering it, and the professional and ethical duties of a software engineer.

Foundations of Software Engineering (Chapter 1)

The Nature of Software

Software is a logical, not physical, element. It doesn't "wear out" like hardware, but it can "deteriorate" as changes and bug fixes are applied over time, degrading its structure and introducing new faults.

Layered Technology Model

Software engineering is a layered technology:

  • A Quality Focus (Bedrock): The foundation is a commitment to quality, such as Total Quality Management (TQM).
  • Think! Clear thought before action produces better results.

Core Process Principles (from Practice and Requirements)

These principles guide the roadmap and flow of a project.

  • Be Agile: A mindset of simplicity and adapting to change.
  • Focus on Quality at Every Step: Quality is an exit condition for every task, not an afterthought.
  • Be Ready to Adapt: The process is a guide, not a rigid set of rules.
  • Build an Effective Team: A team built on trust and respect is the key to success.
  • Establish Mechanisms for Communication: Information must flow freely.
  • Manage Change: Have a defined process for handling inevitable changes.
  • Assess Risk: Identify potential risks early and have contingency plans.
  • Create Work Products That Provide Value: Don't create documents for the sake of it.

Professional, Ethical, and Social Responsibility (Chapter 5, 30)

This is built on three interconnected pillars:

Professional Responsibility

The obligation to deliver high-quality, competent work.

  • Competence: A commitment to lifelong learning to stay current (e.g., not using broken crypto like MD5 for passwords).
  • Quality & Due Diligence: Ensuring Reliability, Security, Usability, and Maintainability. A failure of due diligence can be catastrophic (e.g., the Therac-25 tragedy).
  • Honesty & Integrity: Respecting Intellectual Property (IP) and being truthful about project status.
  • Accountability: Taking ownership of mistakes and using tools like Blameless Post-mortems to fix systems, not blame people.

Ethical Responsibility

The moral compass that guides our decisions, best summarized by the ACM/IEEE-CS Software Engineering Code of Ethics.

  1. Principle 1: PUBLIC: Act consistently with the public interest. This is the highest priority.
  2. Principle 2: CLIENT AND EMPLOYER: Act in the best interests of your client and employer, consistent with the public interest.
  3. Principle 3: PRODUCT: Ensure products meet the highest professional standards.
  4. Principle 4: JUDGMENT: Maintain integrity and independence in your professional judgment (e.g., by disclosing a Conflict of Interest).
  5. Principle 5: MANAGEMENT: Promote an ethical approach to management, including setting realistic goals and fostering psychological safety.
  6. Principle 6: PROFESSION: Advance the integrity and reputation of the profession.
  7. Principle 7: COLLEAGUES: Be fair to and supportive of your colleagues (e.g., giving constructive code reviews).
  8. Principle 8: SELF: Participate in lifelong learning to grow your competence and ethical understanding.

Social Responsibility

Awareness of the broader impact of our work on society.

  • Privacy: Being a good steward of user data. This includes Informed Consent, Data Minimization, and complying with regulations like the GDPR (General Data Protection Regulation).
  • Bias and Fairness: Understanding that algorithms are not automatically objective. Systems trained on biased data will produce biased results (e.g., Amazon's AI recruiting tool).
  • Accessibility (a11y): Designing software that can be used by people with disabilities (e.g., by following WCAG (Web Content Accessibility Guidelines)).
  • Environmental Impact: Recognizing that software runs on hardware that consumes energy. Green Software Engineering aims to build sustainable software.

Topic 2: Software Processes & Standards

This topic covers the "road maps" or frameworks we use to build software.

Generic Process Framework

The five core activities present in all software development.

  1. Communication: Collaborating with the customer to gather requirements.
  2. Planning: Creating the "map" for the project (tasks, risks, schedule).
  3. Modeling: Creating models (like sketches or diagrams) to understand requirements and design a solution.
  4. Construction: Writing code and testing it.
  5. Deployment: Delivering the software to the customer and getting feedback.

Process Models (Chapter 2, 3)

Prescriptive Models

Advocate for an orderly, sequential approach.

  • Waterfall Model: A sequential model where one phase must be completed before the next begins (Communication -> Planning -> Modeling -> Construction -> Deployment).
    • Pros: Simple, enforces discipline, good for projects with stable, well-understood requirements (e.g., safety-critical systems).
    • Cons: Inflexible, cannot handle changing requirements, customer gets no feedback until the very end.

Evolutionary Models

Produce increasingly complete versions of the software iteratively.

  • Prototyping Model: A model where a "mock-up" (prototype) is built quickly, shown to the customer for feedback, and then refined. This is excellent for clarifying uncertain requirements.
  • Spiral Model: Combines the iterative nature of prototyping with the systematic control of the waterfall model. Its defining feature is that it explicitly incorporates risk assessment in each loop of the spiral.

Agile Development (Chapter 3)

A philosophy that emphasizes flexibility, communication, and speed. It values:

  • Individuals and interactions over processes and tools
  • Working software over comprehensive documentation
  • Customer collaboration over contract negotiation
  • Responding to change over following a plan

Agile Frameworks

  • Scrum: An agile framework where work is done in short, time-boxed iterations called Sprints (typically 1-4 weeks).
    • Product Backlog: A prioritized list of all features.
    • Sprint Backlog: The set of features the team commits to building in one sprint.
    • Daily Scrum: A short daily meeting (What I did, What I'll do, What's blocking me).
    • Sprint Review: Demonstrating the working software to stakeholders at the end of the sprint.
  • Kanban: A method focused on visualizing work and maximizing flow. It uses a Kanban Board (e.g., To Do, In Progress, Done) and a key principle of Limiting Work in Progress (WIP) to prevent bottlenecks.
  • DevOps: While not a single framework, DevOps is a culture and set of practices (referenced in Chapter 3.5) that combines software development (Dev) and IT operations (Ops). It aims to shorten the development life cycle and provide continuous delivery with high software quality, heavily relying on automation for CI/CD (Continuous Integration/Continuous Deployment).

Software Process Improvement (Chapter 28)

This is a dedicated activity (referenced in Chapter 28) focused on assessing and improving the maturity and effectiveness of your team's software development processes.

CMMI (Capability Maturity Model Integration)

This is the most well-known process improvement model (referenced in 28.3). It provides a framework for organizations to measure their process maturity against a scale of five levels:

  1. Initial: Processes are chaotic, unpredictable, and reactive.
  2. Managed: Basic project management practices are established.
  3. Defined: Processes are standardized and documented across the organization.
  4. Quantitatively Managed: The organization controls its processes using statistical and quantitative data.
  5. Optimizing: The organization uses data to drive continuous process improvement.

Topic 3: Software Development Phases

This topic details the "how-to" of analysis, design, and construction.

Requirements & Analysis (Chapter 7, 8)

Requirements Engineering

The 7-task process for understanding what needs to be built.

  1. Inception: The start of the project; establishing a basic understanding.
  2. Elicitation: Gathering requirements from stakeholders (this is hard!).
  3. Elaboration: Refining the raw requirements into a detailed model.
  4. Negotiation: Reconciling conflicts and prioritizing requirements (e.g., trading features for time).
  5. Specification: Formally documenting the requirements (e.g., in a Software Requirements Specification (SRS) document).
  6. Validation: Reviewing the specification to ensure it's correct and complete. (Asks: "Are we building the right product?")
  7. Requirements Management: Identifying, controlling, and tracking changes to requirements.

Requirements Modeling

Creating models to understand requirements.

  • Scenario-Based Models: Describe the system from the user's point of view.
    • Use Cases: Tell a "story" of how an actor achieves a goal (e.g., "Withdraw Cash").
    • Actors: A role (person or system) that interacts with the system.
  • Class-Based Models: Describe the "things" (objects) in the system and their relationships.
    • Class Diagrams: Show classes, their attributes (data), and their operations (methods).
  • Behavioral Models: Describe how the system responds to events over time.
    • State Diagrams: Show the different states an object can be in and the events that trigger a transition from one state to another.

Software Design (Chapter 9-14)

Design Concepts

  • Technical Debt: The implied cost of rework caused by choosing an easy, limited solution now instead of a better approach that would take longer.
  • Abstraction: Hiding complexity to expose only essential features (e.g., a "save" button hides the complexity of file I/O).
  • Modularity: Breaking a system into discrete components (modules) to manage complexity.
  • Separation of Concerns (SoC): A design principle for separating a program into distinct sections, each addressing a separate concern (e.g., separating UI logic from business logic).
  • Information Hiding: Hiding the internal implementation details of a module from other modules.
  • Functional Independence: The goal of good design, measured by:
    • Cohesion (High is good): The degree to which the elements inside a module belong together (i.e., it does one thing well).
    • Coupling (Low is good): The degree of interdependence between modules.

Architectural Design

High-level strategic decisions about the system's structure.

  • Data-Centered: A central data store (e.g., a database) that other components interact with.
  • Data-Flow (Pipe and Filter): Data is transformed by a series of components (filters) connected by "pipes."
  • Layered Architecture: Organizes the system into layers, each providing services to the layer above it (e.g., Presentation Layer, Business Logic Layer, Data Access Layer).
  • Model-View-Controller (MVC): A pattern that separates concerns:
    • Model: Manages the data and business logic.
    • View: The user interface that displays the data.
    • Controller: Handles user input and updates the Model and View.

Component-Level Design

Detailed design of individual components, often guided by the SOLID principles:

  • Open-Closed Principle (OCP): A component should be "open for extension, but closed for modification." (You should be ableD to add new features without changing existing code).
  • Liskov Substitution Principle (LSP): Subclasses should be substitutable for their base classes.
  • Interface Segregation Principle (ISP): Many client-specific interfaces are better than one general-purpose interface (don't force a class to implement methods it doesn't need).

User Experience (UX) Design

Designing for the user's entire experience.

Golden Rules of Interface Design:
  1. Place the User in Control: Provide undo, flexible interaction, etc.
  2. Reduce the User's Memory Load: Use defaults, "remember" information, and provide visual cues.
  3. Make the Interface Consistent: Visual and functional consistency allows users to transfer knowledge.

Design Patterns

A general, reusable solution to a commonly occurring problem (e.g., Adapter, Factory, Iterator).

Anti-Patterns

A common "solution" that is ineffective or counterproductive (e.g., The Blob/God Object, Spaghetti Code).

Anti-Patterns

A common "solution" that is ineffective or counterproductive (e.g., The Blob/God Object, Spaghetti Code).

Software Coding & Security Principles (Chapter 6, 18)

Construction Principles

  • Preparation: Understand the design before writing code.
  • Coding: Use structured programming, write clear code.
  • Validation: Conduct code reviews and unit tests.
  • Refactoring: Cleaning up code to improve its quality after it works. This is a key activity in software maintenance.

Secure Coding Principles

  • Minimize Attack Surface: Every new feature (API, form, port) is a potential entry point for an attacker.
  • Validate Input: Never trust user input. This is the #1 rule. Failure leads to attacks like SQL Injection (SQLi). Use parameterized queries (prepared statements) to fix this.
  • Sanitize Output: Clean data before sending it to another system (like a user's browser). This prevents Cross-Site Scripting (XSS).
  • Default Deny: By default, a user should have no permissions. They must be explicitly granted.
  • Principle of Least Privilege: A process or user should only have the bare minimum permissions necessary to do its job.
  • Defense in Depth: Use multiple, overlapping layers of security (e.g., firewall, app logic, encrypted data).
  • Insecure Deserialization: A dangerous vulnerability. Never use unsafe formats like Python's pickle on untrusted data. Use data-only formats like JSON.

Topic 4: Supporting & Cross-Cutting Activities

This topic covers "Umbrella Activities" that are applied throughout the entire software process.

Software Quality Assurance (SQA) (Chapter 15, 16, 17)

SQA is an umbrella activity that defines and conducts activities to ensure the product meets quality standards.

The Quality Assurance Funnel

A typical QA process includes multiple levels of testing:

  1. Unit Testing: Testing individual functions or methods in isolation.
  2. Integration Testing: Testing how different modules interact with each other.
  3. System Testing: Testing the complete, integrated system as a whole to verify it meets requirements.
  4. User Acceptance Testing (UAT): Real users test the software to see if it meets their needs.

Technical Reviews

Assessing work products (like documents or code) to uncover and remove errors early, before they are passed to the next phase.

Software Configuration Management (SCM) (Chapter 22)

SCM is an umbrella activity that manages the effects of change throughout the software process.

Its goal is to answer: What changes were made? Who made them? When were they made? Why were they made?

This includes concepts like:

  • Version Control: Managing different versions of code and other artifacts.
  • Change Management: A defined process for how changes are requested, assessed, approved, and implemented (as seen in the "Smart Fridge" app example).

System Documentation (Chapter 8, 9-14, 21)

Documentation is not a separate phase but a continuous work product.

  • Requirements Documentation: Includes the SRS, Use Cases, and analysis models.
  • Design Documentation: Includes architectural diagrams (e.g., Layered, MVC), Class Diagrams, and State Diagrams.
  • Testing Documentation: Includes test plans and test cases.

Software Maintenance (Chapter 27)

Software maintenance is the process of modifying a software product after it has been delivered to the customer.

  • Software Evolution: Software must change and evolve to remain useful.

Cross-Cutting Security Activities

Microsoft Security Development Lifecycle (SDL)

A formal process for building security into every phase of development.

  • Secure by Design: Security is part of the architecture.
  • Secure by Default: The out-of-the-box configuration is the most secure one (e.g., features off).
  • Secure in Deployment: Provides tools and guides for secure deployment.

Threat Modeling

A structured approach to identifying threats during the design phase.

STRIDE:

A mnemonic for the six threat categories:

  • Spoofing (Impersonating someone)
  • Tampering (Modifying data)
  • Repudiation (Denying an action)
  • Information Disclosure (Exposing data)
  • Denial of Service (Making the system unavailable)
  • Elevation of Privilege (Gaining unauthorized permissions)

Security Risk Analysis

Prioritizing what to fix. A common framework is the NIST Risk Management Framework (RMF).

Security-Focused Testing (Part of SQA)

  • SAST (Static Analysis Security Testing): Tools that scan your source code for known vulnerability patterns.
  • DAST (Dynamic Analysis Security Testing): Tools that attack your running application to find flaws.

Security Maturity Models

Ways to measure and improve your organization's security practices.

  • OWASP SAMM (Software Assurance Maturity Model): A prescriptive model that provides a concrete roadmap for "leveling up" security.
