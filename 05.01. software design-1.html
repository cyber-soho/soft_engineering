Software Design: From Concepts to Components

An Interactive Lecture on Principles, Architecture, and Implementation

Introduction to Software Design

Software design bridges the gap between requirements and implementation. It's where critical decisions are made that impact the software's quality, maintainability, and scalability. A poor design leads to **Technical Debt**: the implied cost of rework caused by choosing an easy solution now instead of a better, more sustainable approach.

Abstraction

The process of hiding complexity and exposing only essential features. It allows us to focus on "what" a thing does, rather than "how" it does it.

Car Ignition System:

You start a car by turning a key. You don't need to know about the starter motor, battery, or fuel injection. The complex process is abstracted into a simple action.

Architecture

The fundamental structures of a software system. It defines high-level components, their relationships, and how they interact to provide conceptual integrity.

House Blueprint:

A blueprint defines a house's structure—room layout, load-bearing walls—without specifying paint colors. It ensures the house is functional and sound, just as software architecture ensures a system is robust and scalable.

Patterns

A general, reusable solution to a commonly occurring problem within a given context. It's a template for how to solve a problem, not a finished piece of code.

A Recipe:

A recipe for a cake is a pattern. It provides ingredients (components) and steps (interactions) to solve the problem of "how to bake a cake." You can reuse the pattern with different ingredients to make different cakes.

Separation of Concerns

A design principle for separating a program into distinct sections, where each section addresses a separate concern (e.g., UI, business logic, data access).

Organizing a Kitchen:

You have separate cabinets for plates (presentation), pots (cooking), and a fridge for food (storage). This separation makes the kitchen easier to manage. Software similarly separates UI logic from business rules.

Modularity

The degree to which a system is composed of discrete components (modules) such that a change to one has minimal impact on others. It's an outcome of Separation of Concerns.

LEGO Bricks:

Each LEGO brick is a self-contained module with a standard interface. You can build complex structures and change one part of your creation without dismantling the whole thing.

Information Hiding

Hiding design decisions in a program that are most likely to change, thus protecting other parts of the program. Implementation details of a module are hidden from other modules.

Using an ATM:

You use a simple interface (screen, keypad) without needing to know the complex internal mechanics or how it connects to the bank's network. This simplifies use and enhances security.

Functional Independence (Cohesion & Coupling)

A measure of good design, achieved by creating modules with high cohesion and low coupling.

  • High Cohesion: Elements inside a module belong together and are focused on a single task.
  • Low Coupling: Modules have minimal interdependence; a change in one does not require changes in others.

Home Entertainment System:

A good system has separate, low-coupled components (player, receiver, speakers). If the player breaks, you replace only the player. A bad, high-coupled system would have everything in one box; if one part breaks, the whole system is useless.


// High Cohesion: Each function has one job.
function validate_email(email, _print) {
    _print("Email validated.");
    return true; // Simplified for demo
}

function save_user_to_db(user, _print) {
    _print("User saved.");
    return true;
}

function send_welcome_email(email, _print) {
    _print("Email sent.");
}

// Low Coupling: This function coordinates calls.
function register_new_user(user, _print) {
    if (validate_email(user.email, _print)) {
        save_user_to_db(user, _print);
        send_welcome_email(user.email, _print);
    }
}

const user = {name: "Alice", email: "alice@example.com"};
register_new_user(user, _print);

Architectural design represents the "big picture" structure of the system. An architectural style is a proven, named collection of design decisions, like a template for a certain category of system.

Data-Centered Architecture

A central data store (e.g., a database) is at the core of the architecture. Other components (clients) access and manipulate this central repository.

University Information System:

A central student database is accessed by independent applications for registration, billing, library services, and online learning. All systems revolve around the shared data.

Data-Flow (Pipe & Filter)

Input data is transformed through a series of components called "filters," with data flowing between them via "pipes." Each filter performs a specific, independent step in a sequence.

Online Photo Printing:

An uploaded photo passes through a sequence of filters: 1. Validate file -> 2. Enhance color -> 3. Resize image -> 4. Send to printer. Each step is a filter in the pipeline.

Layered Architecture

The system is organized into layers, each providing services to the layer above it. This separates concerns, such as UI from business logic. A common example is the three-tier architecture.

Online Pizza Order:

1. Presentation Layer: The website you use to build your pizza. 2. Business Logic Layer: Calculates the price, validates coupons, and finds the nearest store. 3. Data Access Layer: Saves your order to the database.

Model-View-Controller (MVC)

A pattern that divides an application into three interconnected parts to separate internal data representation from the ways information is presented to or accepted from the user.

  • Model: Manages the application's data and business logic.
  • View: Renders the UI and presents data to the user.
  • Controller: Handles user input, interacts with the Model, and updates the View.

Weather App:

The Model holds the weather data. The View is the screen that displays the temperature. The Controller takes your search for a city, tells the Model to fetch new data, and then tells the View to update.


class UserModel { // The MODEL
    constructor() {
        this.users = ['Alice', 'Bob'];
    }
    getUsers() {
        return this.users;
    }
}

class UserView { // The VIEW
    constructor(_print) {
        this._print = _print;
    }
    showUsers(users) {
        this._print(`Users: ${users.join(', ')}`);
    }
}

class UserController { // The CONTROLLER
    constructor(model, view) {
        this.model = model;
        this.view = view;
    }
    updateView() {
        const users = this.model.getUsers();
        this.view.showUsers(users);
    }
}

// --- Execution ---
const model = new UserModel();
const view = new UserView(_print);
const controller = new UserController(model, view);
controller.updateView();

Component-level design focuses on the internal details of each software component. This is where architectural elements are translated into a detailed "blueprint" for coding.

What is a Component?

A modular, deployable, and replaceable part of a system that encapsulates implementation and exposes a set of interfaces. In an OO context, it's a set of collaborating classes. In a traditional context, it's a function or module.

Open-Closed Principle (OCP)

"A component should be open for extension, but closed for modification." This means you should be able to add new functionality without changing existing, tested code. This is usually achieved via abstractions like interfaces.

Payment Processing System:

Instead of one function with many `if` statements for different payment types, you create a `PaymentProcessor` interface. Each payment type (Credit Card, PayPal) is a separate class that implements the interface. To add a new payment type, you create a new class; the existing code doesn't change.


// The "closed for modification" part is this abstract-like class.
class Shape {
    area() {
        throw new Error("This method must be overridden by a subclass.");
    }
}

// The system is "open for extension" by adding new classes.
class Rectangle extends Shape {
    constructor(width, height) {
        super();
        this.width = width;
        this.height = height;
    }
    area() {
        return this.width * this.height;
    }
}

class Circle extends Shape {
    constructor(radius) {
        super();
        this.radius = radius;
    }
    area() {
        return 3.14 * this.radius ** 2;
    }
}

// This calculator never needs to change.
class AreaCalculator {
    calculateArea(shapes, _print) {
        const totalArea = shapes.reduce((sum, shape) => sum + shape.area(), 0);
        _print(`Total Area: ${totalArea}`);
    }
}

const shapes = [new Rectangle(10, 5), new Circle(3)];
const calculator = new AreaCalculator();
calculator.calculateArea(shapes, _print);

Liskov Substitution (LSP)

"Subclasses should be substitutable for their base classes." Code that uses a base class must be able to use any of its subclasses without breaking. The subclass must honor the base class's contract.

Birds That Can't Fly:

If you have a `Bird` class with a `fly()` method, you can't have a `Penguin` subclass inherit from it. A function expecting any `Bird` to fly will break when given a `Penguin`. This violates LSP. The inheritance hierarchy is wrong.

Interface Segregation (ISP)

"Many client-specific interfaces are better than one general-purpose interface." Clients should not be forced to depend on methods they do not use. It's better to have smaller, more focused interfaces.

Smart Printer:

Instead of one big `IMachine` interface with `print()`, `scan()`, and `fax()`, create separate `IPrinter`, `IScanner`, and `IFax` interfaces. A simple printer class only needs to implement `IPrinter`, not methods it can't perform.

Component-Based Dev

An approach that emphasizes building systems by assembling reusable software components, rather than writing everything from scratch. This can speed up development and improve quality.

Building with a CMS:

Building a WordPress site involves assembling components: a core CMS, a theme for design, and plugins for e-commerce, contact forms, and SEO. You build a complex system by integrating pre-built parts.
